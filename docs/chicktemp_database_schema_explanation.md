The ChickTemp database schema is designed using Firebase Realtime Database, a NoSQL cloud database that stores data in a JSON tree structure. Unlike a traditional relational database, Firebase does not use physical table relationships or enforced foreign keys. Instead, the system organizes related records through unique identifiers such as `userId`, `batchId`, `batchKey`, `logId`, and `ticketId`. These identifiers allow the mobile application, ESP32 device, and analytics module to access the correct data while keeping the database flexible and easy to update.

The schema is divided into four main groups: Authentication and Support, Live Sensor and Device Control, Per-User Farm Records, and Analytics, Snapshots, and Local Cache. The Authentication and Support section stores user account information, encoded email references, and support tickets. The `users/{userId}` path contains the user's profile and login-related data, while `users_by_email/{emailKey}` maps an encoded email address to the correct user account. The `support_tickets/{ticketId}` path stores messages submitted through the support feature so user concerns can be recorded and reviewed.

The Live Sensor and Device Control section manages real-time hardware data from the ESP32 system. The `sensor/latest` path stores the most recent temperature, humidity, water level, feeder level, and relay status readings for display in the mobile application. The `environmental_logs/{logId}` path stores historical sensor readings that can be used for monitoring trends and analytics. The `controls/{batchKey}` path stores hardware control states, such as water pump, light bulb, ventilation fan, and feeder servo status, which are written by the application and read by the ESP32 to control the physical devices.

The Per-User Farm Records and Analytics sections store the operational records of each farm user. The `user_data/{userId}/batches/{batchId}` path contains poultry batch information, while related records such as device configurations, mortality records, report records, and analytics summaries are organized under the same user data branch. Even though the schema is shown as grouped database paths rather than connected tables, the records are still related through shared key values such as `userId`, `batchId`, and `batchKey`. This structure keeps account data, farm records, sensor readings, hardware controls, and analytics results organized in separate branches while still allowing the ChickTemp system to connect the correct data across the application.
